# Experience OS 边界确认文档

**版本**：v1.0  
**日期**：2026-06-15  
**用途**：在继续改 PRD、拆文档或进入开发前，先固定 Experience OS 的产品边界、开发边界、比赛表达边界和文档边界。  
**依据**：`01-draft.md`、`000_PRD/99-background.md`、`000_PRD/PRD_Experience_OS.md`、`000_PRD/PRD_Experience_OS_v2.md`，以及三个独立 agent 从产品本质、开发落地、比赛表达三个角度的审视结果。  

---

## 1. 总结论

`PRD_Experience_OS_v2.md` 的方向不是错，而是承载了太多角色。

它同时在做：

- 产品概念说明。
- 开发 PRD。
- 技术设计。
- AI 契约。
- 比赛演示方案。
- 项目计划。
- 文档质量自评。

这导致主文档的中心被稀释，看起来一会儿像为了开发，一会儿像为了比赛，一会儿又像在为项目做自我包装。

后续应把 Experience OS 的主线固定为：

> Experience OS 不是为了记录更多生活，也不是为了包装一个比赛作品，而是用 AI 将日常零散观察和微型试错，沉淀为带条件、可追溯、可反馈、可复用的个人经验规则。

比赛是第一个公开演示场景；开发 PRD 是落地手段；它们都不能取代产品本身。

## 2. 产品本质边界

### 2.1 它真正解决的问题

用户在重复生活场景中不断产生个人化经验，但这些经验通常停留在记忆、聊天、备忘录或零散笔记里，缺少结构、条件、证据和召回机制。

结果是：

- 下一次遇到类似场景，用户仍然要重新回忆。
- 很多经验没有沉淀成可复用规则。
- 用户分不清哪些是稳定规律，哪些只是偶然体验。
- 有价值的生活试错不断流失。

Experience OS 要解决的是：

> 如何让一次生活观察，变成下一次可以直接参考的条件化经验规则。

### 2.2 四个核心对象

| 对象 | 定义 | 示例 |
| --- | --- | --- |
| 观察 | 用户亲自经历或发现的一句话生活经验 | 周末 10 点健身房人少 |
| 条件 | 这条经验成立的上下文 | 周末、上午、该健身房、目标是少排队 |
| 规则 | 从观察中提炼出的可复用判断 | 周末上午 9:45-10:30 适合器械训练 |
| 复用时刻 | 下一次用户遇到相似场景时调出经验 | 周末准备健身时查看已有策略 |

### 2.3 产品核心闭环

```text
观察 -> AI 结构化 -> 形成带条件的经验规则 -> 下次相似场景召回 -> 用户反馈修正
```

这里的关键不是“记录完成”，而是“下次可用”。

### 2.4 产品应该是什么

- 一个个人经验自动结构化系统。
- 一个把微型试错沉淀为经验规则的工具。
- 一个帮助用户保留“下次怎么做更好”的系统。
- 一个用 AI 做后台整理，而不是让用户手动整理的系统。

### 2.5 产品不应该是什么

| 不应该写成 | 原因 |
| --- | --- |
| 生活记录工具 | 生活记录的中心是保存发生过什么；Experience OS 的中心是下次怎么做。 |
| AI 笔记软件 | AI 笔记的中心是整理信息；Experience OS 的中心是提炼个人经验判断。 |
| 泛化生活助手 | 它不替用户做所有生活建议，只处理用户亲自观察到的经验。 |
| 地图产品 | 地图只是某些经验的呈现方式，不是产品本质。 |
| 效率 KPI 或自我管理系统 | 不应把生活变成指标系统，重点是沉淀经验而不是规训行为。 |
| 比赛包装项目 | 比赛是首次演示场，不是产品存在的理由。 |

### 2.6 语言边界

建议使用：

- 个人经验自动结构化。
- 条件化经验规则。
- 可复用行动建议。
- 持续修正个人经验。
- 带条件、带证据、可追溯的规则。

谨慎使用或弱化：

- 决策模型。
- 个人行为优化。
- 生活效率探索者。
- 比赛加分。
- 评委快速理解。
- 综合评分 91/100。
- 个人经验操作系统的过度隐喻。

## 3. 比赛表达边界

### 3.1 比赛的正确角色

比赛是：

- 外部触发器：解释为什么现在做 MVP。
- 截止时间：6 月 30 日前需要可演示。
- 交付形式：需要场景方案、实操演示或作品成品分享。
- 主题匹配依据：属于 AI 生活赋能方向。
- 首次公开演示场：用于验证别人是否能理解这个闭环。

比赛不是：

- 产品北极星。
- 功能优先级的唯一依据。
- 评委取悦工程。
- 把项目包装得更“炫”的理由。

### 3.2 主文档中比赛应该如何出现

主文档可以写：

> 6 月 30 日公司 AI 自动化场景创意比赛是 Experience OS MVP 的首次演示交付场景。

不建议写：

> 本文档是比赛交付型 PRD。

主文档中的比赛内容应控制在：

- 背景事实。
- 截止时间约束。
- 首次演示场景。
- 附录入口。

不应把以下内容放在主文档前部：

- 评委视角。
- 比赛目标。
- 8 分钟演示流程。
- 现场稳定性 100%。
- 加分亮点。
- 比赛话术。
- 详细里程碑。

### 3.3 比赛内容适合放哪里

| 内容 | 应放位置 |
| --- | --- |
| 比赛通知摘要 | 演示文档或附录 |
| 主题匹配说明 | 演示文档 |
| 8 分钟演示流程 | 演示文档 |
| Demo 输入样例 | 开发 PRD 和演示文档均可保留 |
| 现场检查策略 | 演示文档，开发 PRD 保留处理状态说明 |
| 讲解话术 | 演示文档 |
| 评委可能提问 | 演示文档 |
| 比赛里程碑 | 项目计划 |

### 3.4 比赛表达原则

- 产品价值先于比赛价值。
- 真实用户先于评委视角。
- 产品指标和演示指标分开。
- Demo 资产不能写成产品能力。
- 避免“加分项”“亮点包装”“冲奖”这类语言。
- 比赛背景只保留事实，不继承活动通知的宣传语气。
- AI 自动化要落在过程，不落在口号。

## 4. 开发/MVP 边界

### 4.1 本轮开发目标

本轮不是正式商业版本，也不是完整个人知识系统。

本轮目标是：

> H5 优先，使用结构化分析引擎跑通“一句话观察 -> 结构化分析 -> 策略卡片 -> 经验列表 -> 规则库 -> 反馈”的最小闭环。

### 4.2 P0 用户路径

```text
进入首页
-> 输入一句观察
-> 提交
-> 生成观察记录
-> 调用结构化分析
-> 展示策略卡片
-> 保存到经验列表
-> 有可行动建议则进入规则库
-> 用户可查看规则库
-> 用户可标记有用/待观察/不准确
```

### 4.3 P0 必做

| 模块 | P0 边界 |
| --- | --- |
| 运行端 | H5 可运行，手机视口优先 |
| 输入 | 一句话输入、提交、清空、空输入提示 |
| Demo 样例 | 内置 5-8 条样例，可一键填入或展示 |
| AI 服务 | 封装 `analyzeObservation(text)`，输出必须通过结构校验；外部模型输出必须经过统一合约校验后才能进入规则库 |
| 策略卡片 | 展示分类、标题、核心结论、推荐行动、适用条件、注意事项、可复用度 |
| 经验列表 | 按时间倒序展示原文、分类、摘要、创建时间、处理状态 |
| 规则库 | 只展示可复用规则，支持基础分类筛选 |
| 本地持久化 | 刷新后观察和规则不丢 |
| 失败处理 | 结构化校验失败时保留原文，不生成规则卡 |
| 反馈 | 可标记有用/待观察/不准确；P0 只保存状态 |
| 构建 | `npm run build:h5` 应可通过 |

### 4.4 P1 增强

- 接入真实模型适配器，但页面和 store 仍只调用统一分析入口。
- AI 输出字段、结果方向、置信度和失败降级校验。
- 简单相似记录合并：分类 + 关键词 + 时间词。
- 规则证据数量累加。
- 反馈影响规则状态。
- 关键词搜索观察和规则。
- 处理失败后的重试入口。
- 微信小程序构建验证。
- 简单编辑/删除记录。
- 完整 loading、empty、error 状态。
- 规则库排序。

### 4.5 P2 后续方向

- 地点聚合页。
- 时间轴页。
- 静态经验地图概念展示。
- 真实地图 SDK。
- embedding 相似度。
- 多端同步。
- 账号体系。
- 后端数据库。
- 通知提醒。
- 趋势分析。
- 自动推荐。
- 复杂知识库层级。
- 分享/社区功能。

#### 4.5.1 经验地图规划边界

经验地图可以写进产品规划，但它不是 MVP 核心能力，也不是地图导航产品。

它的定位是：

> 当经验中包含地点信息时，系统按地点聚合相关观察和规则，形成用户自己的“地点经验视图”。

可规划的能力：

- 按地点聚合经验规则。
- 展示某个地点下的历史观察、策略卡片和注意事项。
- 帮助用户回看“我在这个地方积累过什么经验”。
- 后续可从地点列表、地点卡片升级为真实地图视图。

不应写成：

- 查找附近地点。
- 地图导航。
- 本地生活攻略。
- 店铺推荐平台。
- 真实地图 SDK 的 MVP 依赖。

第一阶段可以先使用“地点列表/地点卡片”表达经验地图，不阻塞 MVP。

#### 4.5.2 时间轴规划边界

时间轴可以写进产品规划，但它不是长日志，也不是日记系统。

它的定位是：

> 按时间展示经验从单次观察、AI 提炼、用户反馈到规则修正的演化过程。

可规划的能力：

- 按日期展示用户记录过的观察。
- 展示一条规则背后的多次观察证据。
- 展示规则如何被用户反馈修正。
- 帮助用户理解某条经验是如何从“待观察”变成“可复用规则”的。

不应写成：

- 每日生活流水账。
- 情绪日记。
- 长篇日志编辑器。
- 普通笔记时间线。

第一阶段可以先在规则详情中展示“证据记录/更新时间”，后续再升级为完整时间轴。

### 4.6 明确不做

- 长日志编辑器。
- 手动复杂标签体系。
- 账号登录。
- 云同步。
- 社交分享。
- 真实地图 SDK 作为 MVP 依赖。
- 医疗、法律、财务类高风险建议。
- 完整 AI 记忆系统。
- 多用户数据权限。
- 商业化付费能力。
- 自动推送提醒。

### 4.7 几个关键简化决定

| 项目 | 边界决定 |
| --- | --- |
| 真实 AI | P0 使用结构化分析引擎，后续接真实模型适配器 |
| 去重关联 | P0 做标题、分类、地点的轻量相似合并 |
| 反馈优化 | P0 只保存反馈状态，P1 再影响规则 |
| 置信度 | P0 展示高/中/低/待观察，不展示伪精确小数 |
| 地图 | P2，不阻塞 MVP |
| 微信小程序 | 兼容方向，P0 以 H5 为准 |

## 5. 文档边界

### 5.1 不再维护一个超长 PRD

后续不建议继续把所有内容塞进 `PRD_Experience_OS_v2.md`。

建议将文档拆成 6 份，每份服务一个读者和一个决策场景。

```text
000_PRD/
  CONCEPT_Experience_OS.md
  DEV_PRD_Experience_OS_MVP.md
  TECH_Design_Experience_OS.md
  AI_Contract_Experience_OS.md
  DEMO_Experience_OS.md
  PLAN_Experience_OS.md
```

### 5.2 文档职责

| 文档 | 回答的问题 | 不放什么 |
| --- | --- | --- |
| `CONCEPT_Experience_OS.md` | 为什么做、给谁用、产品本质是什么 | 验收标准、字段、目录、比赛脚本 |
| `DEV_PRD_Experience_OS_MVP.md` | 本轮做什么、怎么验收、什么不做 | 产品长愿景、比赛话术 |
| `TECH_Design_Experience_OS.md` | 怎么实现、目录怎么分、状态怎么管 | 产品叙事、演示稿 |
| `AI_Contract_Experience_OS.md` | AI 输入输出、结构校验、规则边界 | 页面排期、比赛讲解 |
| `DEMO_Experience_OS.md` | 怎么演示、怎么讲、怎么检查现场状态 | 开发字段细节 |
| `PLAN_Experience_OS.md` | 排期、风险、待确认、决策记录 | 产品主叙事 |

### 5.3 现有 v2 文档处理建议

`PRD_Experience_OS_v2.md` 可以保留为“综合草稿”或拆分来源，不建议继续作为唯一主文档。

处理方式：

- 不立即删除。
- 后续拆文档时从 v2 提取内容。
- 拆完后将 v2 标注为历史版本或移入 `archive/`。

| 编号 | 决定 |

| D02 | 比赛是首次演示场景，不是产品北极星。 |

## 8. 一句话边界

> Experience OS 的主线是“个人经验如何被 AI 自动结构化并在下一次相似场景中复用”；开发文档负责把它做出来，比赛文档负责把它讲清楚，但二者都不应该改写产品本身。
