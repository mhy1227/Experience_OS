## 2. 执行摘要

Experience OS 是一个面向个人生活场景的 AI 自动化产品。用户只需要用一句话记录日常观察，例如“11 点面包最好吃”“周末 10 点健身房人少”，系统自动完成分类、规律提炼、去重关联，并输出可复用的策略卡片。

它解决的问题不是“如何记录更多内容”，而是“如何让生活经验自动变成下次可以直接使用的决策依据”。对于比赛而言，该项目能直观展示 AI 自动化从自然语言输入到结构化输出的完整闭环，适合作为 AI 生活赋能方向的实操作品。

MVP 以 H5 或小程序形态呈现，核心交付是一条观察从输入到生成策略卡片、进入规则库、支持反馈优化的完整链路。地图和时间轴作为增强亮点，可根据开发进度选择静态演示或二期实现。

## 3. 背景与机会

### 3.1 比赛背景

公司将在 6 月 30 日下午举行 AI 自动化场景创意比赛，主题包括 AI 工作赋能与 AI 生活赋能。参赛要求是围绕 AI 自动化完成场景方案、实操演示或作品成品分享，内容原创、符合主题即可。

Experience OS 选择“AI 生活赋能”方向，原因是生活经验高度碎片化，天然适合用 AI 做自动分类、总结和策略生成；同时该场景演示成本低，观众容易理解，现场输入一条观察即可看到 AI 自动化结果。

### 3.2 用户背景

人在日常生活中持续进行大量微型实验：

- 什么时间去某家店体验最好。
- 哪种路线更省时间。
- 哪种消费选择更稳定。
- 哪种习惯让自己状态更好。
- 哪些地点、时间、行为组合更容易得到好结果。

这些经验通常以记忆、聊天、备忘录或零散笔记存在。它们很少被结构化，也很少在下一次决策时被主动召回。

### 3.3 产品机会

AI 的价值不只是“总结一篇长文”，还可以成为个人经验的后台整理员：

- 用户用自然语言记录，不需要学习复杂模板。
- AI 将观察转为分类、条件、结论、建议和置信度。
- 系统逐步形成规则库，让经验可检索、可复用、可更新。
- 当类似场景再次出现时，用户可以直接参考既有策略，而不是重新试错。

## 4. 产品定义

### 4.1 一句话定义

一个用 AI 将日常零散生活观察与行为实验自动沉淀为可复用经验规则的个人经验系统。

### 4.2 产品本质

**不是**：

- 长日志工具。
- 备忘录工具。
- AI 聊天工具。
- 手动分类的笔记软件。

**而是**：

- 将非结构化观察转为结构化经验。
- 将一次性记录转为可复用规则。
- 将个人试错过程沉淀为可持续优化的行为策略库。

### 4.3 核心闭环

```text
观察 -> 一句话记录 -> AI 结构化 -> 生成策略卡片 -> 加入规则库 -> 下次使用 -> 反馈修正
```

### 4.4 核心价值主张

| 价值 | 描述 |
| --- | --- |
| 低成本记录 | 用户只写一句话，不填写复杂表单。 |
| 自动结构化 | AI 自动识别分类、时间、地点、对象、条件和结论。 |
| 可行动输出 | 输出不是摘要，而是下次可执行的建议。 |
| 持续复用 | 规则库积累后，用户可以按场景快速查找历史经验。 |
| 反馈优化 | 用户确认有效或无效后，规则置信度和适用条件持续更新。 |

## 5. 目标与指标

### 5.1 比赛目标

| 目标 | 说明 |
| --- | --- |
| 完成一个可演示的 AI 自动化闭环 | 从输入一句观察到输出结构化策略卡片。 |
| 展示 AI 生活赋能价值 | 让评委快速理解“经验自动沉淀”的实际意义。 |
| 体现原创性和可扩展性 | 不停留在聊天总结，而是形成个人经验操作系统概念。 |
| 控制交付复杂度 | MVP 聚焦输入、AI 处理、规则卡片、规则库，不把地图作为硬依赖。 |

### 5.2 用户价值目标

| 目标 | 衡量方式 |
| --- | --- |
| 降低记录成本 | 单条记录输入时间 <= 10 秒，文本建议 10-50 字。 |
| 提升经验复用能力 | 每条有效记录至少生成 1 条可执行建议或标记为“待观察”。 |
| 降低重复试错 | 相似记录能被识别并合并到已有规则。 |
| 提升决策信心 | 规则卡片展示结论、适用条件、注意事项和可复用度。 |

### 5.3 MVP 成功指标

| 指标 | 目标值 | 说明 |
| --- | ---: | --- |
| 结构化成功率 | >= 80% | Demo 样例中，AI 能正确抽取分类、结论、条件。 |
| 规则可行动率 | >= 70% | 生成结果能表达明确建议，而非泛泛总结。 |
| 单条处理耗时 | <= 3 秒 | 分析过程必须明确显示处理中状态，超时按失败处理。 |
| 用户操作步骤 | <= 3 步 | 输入、提交、查看结果。 |
| 本地可演示稳定性 | 100% | 现场准备固定样例与检查清单，避免临时操作影响演示节奏。 |

## 6. 用户与场景

### 6.1 用户角色

| 角色 | 说明 | 关注点 |
| --- | --- | --- |
| 主要用户：生活效率探索者 | 对时间、消费、习惯、路线、体验优化敏感的人 | 希望少试错、少遗忘、下次直接用经验 |
| 比赛观众/评委 | 现场观看 AI 自动化场景的人 | 是否有清晰痛点、是否体现 AI、是否可演示 |
| 未来扩展用户：个人知识管理用户 | 已经使用笔记或效率工具的人 | 是否比普通记录更轻、更自动、更可复用 |

### 6.2 典型场景

| 场景 | 原始观察 | 期望输出 |
| --- | --- | --- |
| 饮食 | 11 点去楼下面包店，刚出炉的吐司最好吃 | 建议 10:50-11:20 购买吐司，工作日有效，可复用度高 |
| 健身 | 周末 10 点健身房人少，器械不用排队 | 周末 9:45-10:30 适合力量训练，人流少 |
| 出行 | 下雨天走 B 口到公司更近，还不用绕路 | 雨天通勤优先选择 B 口，减少绕行 |
| 购物 | 猫碗浅口的比深口的更适合，猫吃得更干净 | 猫用品优先选择浅口宽边碗，适合当前宠物偏好 |
| 办公 | 下午 3 点写方案容易卡，上午 10 点效率最高 | 复杂写作任务优先安排上午 10 点前后 |

### 6.3 用户旅程

1. 用户在生活中发现一个小经验。
2. 打开 Experience OS。
3. 输入一句自然语言观察。
4. 系统展示 AI 分析过程或处理中状态。
5. 系统生成策略卡片。
6. 用户确认“有用/待观察/不准确”。
7. 有用规则进入规则库。
8. 下一次遇到类似场景时，用户按分类、关键词、地点或时间查找规则。

## 7. MVP 范围

### 7.1 Phase 1：比赛 MVP 必做

| 模块 | 功能 | 优先级 |
| --- | --- | --- |
| 一句话输入 | 输入生活观察，支持提交和清空 | P0 |
| AI 结构化 | 自动识别分类、关键词、时间、地点、对象、结论 | P0 |
| 策略卡片 | 展示核心结论、建议行动、成功条件、注意事项、可复用度 | P0 |
| 经验列表 | 展示原始记录、分类、AI 摘要和生成时间 | P0 |
| 规则库 | 展示已沉淀的可复用规则，支持按分类筛选 | P0 |
| 反馈按钮 | 用户可标记有用、待观察、不准确 | P1 |
| Demo 样例 | 内置 5-8 条生活样例，确保现场可演示 | P0 |

### 7.2 Phase 2：比赛加分或后续增强

| 模块 | 功能 | 优先级 |
| --- | --- | --- |
| 经验地图 | 按地点聚合经验，展示地点相关规则 | P2 |
| 时间轴 | 按日期和时间段查看经验积累 | P2 |
| 相似记录合并 | 多条类似观察合并为同一规则，并提升置信度 | P1 |
| 检索 | 按关键词搜索观察和规则 | P1 |
| 多端同步 | H5 和小程序之间同步数据 | P3 |

### 7.3 明确不做

- 不做长篇日记编辑器。
- 不做复杂知识库层级管理。
- 不做手动标签体系作为核心流程。
- 不做社交分享和社区功能。
- 不做财务、医疗、法律等高风险决策建议。
- MVP 不强依赖真实地图 SDK。
- MVP 不要求复杂账号体系。

## 8. 功能需求

### 8.1 FR01 一句话记录

**用户故事**：  
作为生活效率探索者，我希望用一句话快速记录观察，这样我不用在记录时被表单打断。

**需求说明**：

- 首页提供明显输入框。
- 支持 10-100 字自然语言输入；建议文案引导用户写 10-50 字。
- 输入为空时不可提交。
- 输入过长时提示精简，不阻断本地保存，但 AI 处理可提示“信息过长，建议拆分”。
- 提交后生成一条观察记录。

**验收标准**：

- 输入“周末 10 点健身房人少，器械不用排队”后，可生成一条原始观察。
- 空输入点击提交时，应提示“先写一条观察”。
- 提交成功后输入框清空，并显示处理中或结果卡片。

### 8.2 FR02 AI 分类

**用户故事**：  
作为用户，我希望系统自动判断观察属于什么类型，这样我不用手动整理。

**需求说明**：

- 默认分类包括：饮食、购物、出行、运动、工作、生活、偏好、时间、地点、其他。
- AI 输出一个主分类，可附带 1-3 个标签。
- 分类置信度低时归入“待确认”或“其他”。

**验收标准**：

- “11 点面包最好吃”应优先归类为“饮食”，标签包含“时间”“面包”。
- “周末 10 点健身房人少”应优先归类为“运动”，标签包含“周末”“人流”。
- 无法判断的输入应进入“其他”，不能生成明显错误分类。

### 8.3 FR03 规律提炼

**用户故事**：  
作为用户，我希望系统从观察中提炼出下次可用的规律，这样记录不只是存档。

**需求说明**：

AI 需要尽量抽取：

- 行为对象：用户做了什么。
- 时间条件：什么时候有效。
- 地点条件：在哪里有效。
- 影响因素：导致体验好或差的原因。
- 核心结论：这条观察说明了什么。
- 推荐行动：下次应怎么做。
- 注意事项：什么时候可能不适用。

**验收标准**：

- 输入“工作日上午 11 点去楼下面包店，吐司刚出炉最好吃”，输出应包含“工作日上午 10:50-11:20 可优先购买吐司”这类行动建议。
- 若输入只包含感受，如“今天这家店还不错”，系统应生成“待观察”而不是强行输出高置信规则。
- 每张策略卡至少包含“核心结论”和“推荐行动”。

### 8.4 FR04 去重与关联

**用户故事**：  
作为用户，我希望类似经验能自动合并，这样规则库不会越来越乱。

**需求说明**：

- 新观察提交后，与历史观察和规则做相似度判断。
- 若与已有规则高度相似，则更新规则的证据数、最近验证时间和置信度。
- 若相似但条件冲突，则保留为关联观察，提示“可能存在条件差异”。
- MVP 可使用简单关键词和分类匹配模拟相似度；后续可接入 embedding。

**验收标准**：

- 连续输入两条关于“11 点面包刚出炉”的观察，应合并或关联到同一条面包购买规则。
- 输入“周末 11 点面包卖完了”时，不应简单提升原规则置信度，而应记录为限制条件或冲突证据。

### 8.5 FR05 策略卡片

**用户故事**：  
作为用户，我希望看到一张清晰卡片，直接告诉我经验结论和下次行动。

**需求说明**：

策略卡片字段：

- 分类标签。
- 规则标题。
- 核心结论。
- 推荐行动。
- 适用条件。
- 注意事项。
- 证据来源。
- 可复用度：高/中/低/待观察。
- 用户反馈状态：有用/待观察/不准确。

**验收标准**：

- 卡片无需阅读原始记录，也能理解下次怎么做。
- 卡片应保留原始观察入口，方便用户追溯。
- 可复用度低的卡片应明显标记“待观察”，避免误导。

### 8.6 FR06 经验列表

**用户故事**：  
作为用户，我希望能看到所有原始观察和 AI 总结，这样我可以回看自己记录过什么。

**需求说明**：

- 按时间倒序展示观察记录。
- 每条记录显示原文、分类、AI 摘要、创建时间、处理状态。
- 支持基础分类筛选。
- 处理失败时显示失败状态和重试入口。

**验收标准**：

- 新提交记录应立即出现在列表顶部。
- 处理失败不应丢失原文。
- 切换分类筛选后，只展示对应分类记录。

### 8.7 FR07 规则库

**用户故事**：  
作为用户，我希望可复用经验沉淀到规则库，这样下次可以直接查找。

**需求说明**：

- 规则库只展示具有复用价值的规则。
- 支持按分类查看。
- 展示规则标题、推荐行动、可复用度、证据数量、最近更新时间。
- 用户反馈“不准确”的规则应下调可复用度或进入待确认。

**验收标准**：

- “待观察”记录可以存在于经验列表，但不一定进入规则库。
- 规则库中的每条规则必须有明确推荐行动。
- 当用户标记“不准确”后，规则可复用度应下降或展示待修正状态。

### 8.8 FR08 反馈优化

**用户故事**：  
作为用户，我希望告诉系统这条规则是否有用，这样系统可以持续变准。

**需求说明**：

- 每张策略卡提供三个反馈选项：有用、待观察、不准确。
- 有用：提升可复用度或证据权重。
- 待观察：保留但不提升置信度。
- 不准确：降低可复用度，提示需要更多证据。

**验收标准**：

- 点击反馈后，卡片状态立即更新。
- 反馈记录应保存到本地状态。
- 不准确反馈不能删除原始记录。

## 9. AI 处理规则

### 9.1 输入输出结构

**输入**：

```json
{
  "text": "周末10点健身房人少，器械不用排队",
  "createdAt": "2026-06-15T10:00:00+08:00",
  "optionalContext": {
    "location": "健身房",
    "source": "manual"
  }
}
```

**输出**：

```json
{
  "category": "运动",
  "tags": ["周末", "10点", "人流", "器械"],
  "summary": "周末10点健身房人流较少，适合训练。",
  "ruleTitle": "周末上午健身房低峰训练策略",
  "conclusion": "周末10点附近健身房人少，器械等待时间低。",
  "recommendation": "周末优先在9:45-10:30到达健身房进行器械训练。",
  "conditions": ["周末", "上午10点前后", "目标是减少排队"],
  "warnings": ["节假日或促销活动可能导致人流变化"],
  "reusability": "高",
  "confidence": 0.82,
  "status": "active"
}
```

### 9.2 规则生成原则

| 原则 | 说明 |
| --- | --- |
| 不夸大 | 单条观察不能包装成绝对规律，只能表达为“建议/可能/待观察”。 |
| 可执行 | 推荐行动必须包含清晰动作，例如“优先在 10:50-11:20 购买”。 |
| 保留条件 | 时间、地点、天气、对象等条件必须尽量保留。 |
| 标记不确定性 | 证据不足时输出“待观察”，不进入高可复用规则。 |
| 可追溯 | 每条规则能追溯到原始观察。 |

### 9.3 可复用度判定

| 等级 | 判定条件 | 展示策略 |
| --- | --- | --- |
| 高 | 条件清晰、行动明确、已有多条相似证据或用户确认有用 | 进入规则库重点展示 |
| 中 | 条件较清晰，但证据较少 | 进入规则库，提示继续观察 |
| 低 | 行动建议不够稳定或条件缺失 | 保留在经验列表 |
| 待观察 | 原始输入模糊、仅有感受、无法形成行动建议 | 不作为正式规则 |

### 9.4 严格处理方案

若 AI 接口不可用或响应超时：

- 保留原始观察。
- 使用本地规则生成基础分类和模板化卡片。
- 标记“AI 分析待补充”。
- 比赛 Demo 准备固定样例输出，确保现场演示不中断。

## 11. 页面与交互

### 11.1 首页：快速记录 + 最新策略

**目标**：让用户打开后立刻记录，提交后立刻看到 AI 输出。

**核心元素**：

- 一句话输入框。
- 提交按钮。
- 示例输入快捷项。
- 最新生成的策略卡片。
- 入口：经验列表、规则库。

**交互要求**：

- 输入区应始终是页面第一优先级。
- 提交后不要跳转复杂页面，优先在当前页展示结果。
- 处理状态需要可见，例如“正在提炼经验规则”。

### 11.2 经验列表页

**目标**：保留所有观察，提供回看与追溯。

**核心元素**：

- 分类筛选。
- 记录卡片。
- 处理状态。
- 原文和摘要。

### 11.3 规则库页

**目标**：集中展示真正可复用的经验策略。

**核心元素**：

- 分类导航。
- 规则卡片。
- 可复用度。
- 证据数量。
- 最近更新时间。

### 11.4 经验地图页（扩展）

**目标**：作为比赛亮点展示地点经验积累。

**MVP 处理建议**：

- 不强依赖真实地图 SDK。
- 可先使用“地点卡片墙”或“地点列表 + 经验数量”模拟地图价值。
- 若时间充足，再接入地图组件。

## 12. 非功能需求

| 编号 | 需求 | 标准 |
| --- | --- | --- |
| NFR01 | 轻量输入 | 用户无需填写结构化表单即可完成记录。 |
| NFR02 | 响应速度 | 普通 AI 处理目标 <= 3 秒；超过 1 秒显示处理中状态。 |
| NFR03 | 本地可用 | Demo 数据和历史记录优先本地保存，避免现场依赖后端。 |
| NFR04 | 数据安全 | 不展示敏感个人信息；默认不做公开分享。 |
| NFR05 | 可解释 | AI 结论需要展示依据、条件和不确定性。 |
| NFR06 | 可扩展 | 数据结构支持后续增加地图、时间轴、搜索和多端同步。 |
| NFR07 | 跨端适配 | 优先适配 H5；界面结构兼容微信小程序。 |

## 13. 技术约束与实现建议

### 13.1 当前技术栈

| 层级 | 建议 |
| --- | --- |
| 前端框架 | uni-app + Vue 3 |
| 状态管理 | Pinia |
| 样式 | Sass |
| 运行目标 | H5 优先，兼容微信小程序 |
| 数据存储 | MVP 使用本地 storage；后续可扩展后端或云数据库 |
| AI 接入 | 封装 `analyzeObservation(text)` 服务，输出必须通过结构校验 |

### 13.2 推荐目录思路

```text
src/
  pages/
    index/              # 快速记录与最新卡片
    records/            # 经验列表
    rules/              # 规则库
  services/
    aiAnalyzer.ts       # 结构化分析服务
    storage.ts          # 本地存储
  stores/
    experience.ts       # 观察和规则状态
  types/
    experience.ts       # Observation / ExperienceRule 类型
  styles/
    tokens.scss         # 基础视觉变量
```

### 13.3 AI 服务接口边界

- 页面不直接调用具体 AI SDK。
- 页面只调用业务服务：`submitObservation(text)`。
- 业务服务负责保存原始记录、调用 AI、更新规则、处理失败。
- AI 输出必须经过字段校验，缺失关键字段时标记处理失败。

## 15. 风险与应对

| 风险 | 概率 | 影响 | 应对 |
| --- | --- | --- | --- |
| AI 输出不稳定 | 中 | 高 | 固定输出 schema；输出后做字段校验，不合格不生成规则。 |
| 现场网络不可用 | 中 | 高 | 支持本地模拟分析；提前缓存样例输出。 |
| 规则生成过度自信 | 中 | 中 | 引入可复用度、置信度和“待观察”状态。 |
| 地图功能开发超时 | 中 | 中 | 地图简化为地点聚合列表，不影响 MVP 主流程。 |
| 产品被误解为笔记工具 | 中 | 中 | 演示重点放在“AI 生成可行动规则”和“规则库复用”。 |
| 输入样例太弱 | 低 | 中 | 预置能体现时间、地点、行为、结果的样例。 |
