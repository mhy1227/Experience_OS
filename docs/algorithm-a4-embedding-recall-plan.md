# A4 实施方案:端上 embedding + 语义召回

> 日期:2026-06-23 · 状态:实施方案(纯方案,未编码)· ⚠️ **重 + 吃数据,优先级低于必需链**
> 关联:`docs/algorithm-upgrade-plan.md`(A4)、`services/decisionHints.ts`(现关键词召回)、`docs/algorithm-evaluation-and-data-plan.md`(冷启动/融合)、`docs/backend-roadmap.md` B3(服务端向量,本文主张优先放端上)
> 目标:决策召回从"对字"升级到"对意思"——**浏览器里跑 embedding,向量不出设备**。

---

## 0. 修的痛 + 诚实定位
- **痛**:TF-IDF/关键词只能匹配**有相同词**的句子;"没开对齐会" 和 "需求没同步" 一个相同词都没有,但意思接近,现在召回不到。
- **诚实**:这是**整套里最重的一块**(模型体积、加载、性能),且**吃数据**。建议**排在必需链(A1/A2/A3)之后**,数据与体积预算到位再做。

## 1. 方案:端上 embedding
- 用 **transformers.js**(ONNX/WASM)在浏览器加载**小型多语 embedding 模型**(如 `bge-small-zh` / MiniLM 量化版);
- 每条观察/规则 → 定长向量;查询 → 向量 → **余弦 top-k**。

## 2. 步骤
1. **懒加载**模型(首次用到才下载 + 缓存);
2. 写入观察时,后台算 embedding,存 **IndexedDB**(key = 观察 id + 内容 hash,内容变才重算);
3. 决策召回:查询文本 → embedding → 与库内向量算余弦 → top-k。

## 3. 性能 / 体积预算(关键约束)
- 模型几十 MB → **懒加载 + 浏览器缓存**,绝不进主 bundle 关键路径;
- embedding 计算放 **Web Worker**,不阻塞 UI;
- 量化模型降体积/提速;
- 首次加载给明确进度提示。

## 4. 冷启动 / 降级
- embedding **不需要 IDF 语料**,小数据也能算向量(比 TF-IDF 对小数据更友好);
- 但**模型加载成本高** → 建议**用户可选开启**或仅在 N 达阈值时启用(见 eval-plan §4);
- **降级链**:模型加载失败 / 不支持 WASM → 回退 TF-IDF(A2)→ 再回退关键词。全程不崩。

## 5. 存储
- 向量缓存:IndexedDB(localStorage 存不下大量浮点向量);
- 失效策略:内容 hash 变 → 重算;清空数据时一并清向量。

## 6. 接入
- 主接入:`decisionHints` 召回(关键词 → 语义);
- 可选:embedding + DBSCAN 作 A3 聚类的**更强特征**(替代 TF-IDF 向量)。

## 7. 测试设计
1. **同义不同词**:构造意思近、无共词的两句 → 余弦高、能互召回;
2. 余弦正确性(相同向量=1);
3. 缓存命中:同内容不重算;
4. **降级**:模拟加载失败 → 回退 TF-IDF/关键词,不抛错;
5. 清空数据 → 向量库同步清空。

## 8. 选型(面试可讲)
- **为什么端上不服务端?** 向量/原文不出设备 = 真·本地优先 + 隐私;呼应 backend-roadmap"端上做不动再上服务端"。
- **为什么 embedding 不只 TF-IDF?** TF-IDF 对字面;embedding 对语义(同义/复述)。代价是模型体积——按需启用。
- **为什么小模型/量化?** 浏览器环境对体积和速度敏感,小量化模型性价比最高。

## 9. YAGNI
不做:微调/训练 embedding、服务端向量库、跨设备同步向量、ANN 加速(数据多了才上,见 A5)。先把"端上 embedding + 余弦 top-k + 降级"跑通。

---

> 一句话:**A4 让决策召回"按意思找"——浏览器里跑小 embedding、向量存 IndexedDB、不出设备;但它最重最吃数据,排在必需链之后、预算到位再做。**
