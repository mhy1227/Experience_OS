# A6 实施方案:隐私脱敏(Luhn + 香农熵 + 正则)

> 日期:2026-06-23 · 状态:✅ 已实现(2026-06-24,浏览器移植版;见 `docs/superpowers/specs/2026-06-24-a6-privacy-redaction-design.md`)
> 关联:`docs/algorithm-upgrade-plan.md`(A6)、`docs/privacy-status.md`(补的就是这个洞)、`docs/backend-roadmap.md` §1.1(方案 A:移植到前端)
> 目标:**发请求给云端模型前,在浏览器里剥离 PII / 密钥**,标识符根本不出设备。

---

## 0. 为什么(挂核心价值,不只是合规)
现状(`privacy-status.md`):AI 提炼时观察原文**裸发**云端,无脱敏。而最值钱的决策数据恰是敏感的。**前端脱敏 = 让用户敢写敏感决策的前提**,是核心价值路径,不是附属。

## 1. 检测项

| 类型 | 方法 |
|------|------|
| 邮箱 | 正则 |
| 手机号 | 正则(中国 `1[3-9]\d{9}`) |
| 身份证 | 18 位正则 + 校验位 |
| **银行卡** | **Luhn 算法**(mod-10 校验,判真卡号) |
| IP | 正则 |
| **密钥 / API Key** | **香农熵** + 上下文(gitleaks 风格关键词:`key/token/secret=` 后的高熵串) |

## 2. Luhn 算法(银行卡校验)
判断一串数字是不是合法卡号,避免把普通长数字误当卡号:
1. 从右往左,**偶数位**(第 2、4… 位)数字 ×2,超 9 则减 9;
2. 所有位求和;
3. 和能被 10 整除 → 合法卡号。

> 用途:正则先粗筛"13~19 位连续数字",再用 Luhn 确认,**降误报**。

## 3. 香农熵(密钥检测)
量字符串有多"乱":`H = −Σ p(c)·log₂ p(c)`(p 为各字符出现频率)。
- 正常词/句:熵低;
- 随机密钥(如 `sk-a8Df9...`):熵高(经验阈值 ~3.5–4.0 bit/char 以上)。
- 结合**上下文关键词**(`key`/`token`/`secret`/`=` 后)减少误报。

## 4. 替换策略
命中 → 替换为**不可逆占位符**:`[EMAIL]` `[PHONE]` `[ID]` `[CARD]` `[SECRET]` `[IP]`。
- 不可逆(不保留映射);
- 占位符稳定(同类同符),不破坏模型对句子结构的理解。

## 5. 模块 + 接入
- `src/services/redact.ts`(纯函数):`redact(text): { clean: string, hits: RedactHit[] }`;
- **接入点**:模型 client 发请求前(`resilientAnalysis` / `modelClient` 的 send 路径)统一过一道;
- **红线**:必须**客户端**执行(本地优先);纯本地引擎路径本就不外发,无需脱敏。
- UI 可选:提示"已自动打码 N 处敏感信息"。

## 6. 测试设计(node:assert,登记 package.json)
新建 `tests/redact.test.ts`:
1. 各类型分别命中并替换;
2. **Luhn**:合法卡号命中、随机 16 位数字(非法)不误报;
3. **熵**:高熵密钥命中、正常长单词不误报;
4. 占位符不可逆、稳定;
5. 正常句子零误报(防过度打码);
6. 幂等:脱敏后再脱敏不变;
7. 性能:长文本毫秒级。

## 7. 选型(面试可讲)
- **为什么正则+熵不用 ML NER?** 轻量、零依赖、毫秒级、可离线、可解释;敏感标识符多是结构化的,正则+校验+熵已覆盖主流。
- **为什么要 Luhn / 熵而不只正则?** 正则只管"长得像",Luhn/熵管"是不是真的",**降误报**——误报会把正常内容打码、破坏分析。
- **为什么前端不靠中转?** 标识符**根本不出设备**才是真·本地优先;靠中转是信任前移。

## 8. YAGNI
不做:可逆脱敏金库、ML 命名实体识别、自定义规则 UI、跨语言 PII。先把 Luhn+熵+主流正则这套经典、可测、能讲的落地。

---

> 一句话:**A6 在发请求前于浏览器剥离 PII/密钥(Luhn 验卡号、熵揪密钥、正则抓常见 PII),补上 `privacy-status` 那个"原文裸发"的洞——而且它是"让用户敢写敏感决策"的核心前提,不是合规附属。**
