## 1. 实测依据

- 在 `src/` 全量检索 `privacy / redact / 脱敏 / mask / PII / sanitize / scrub` —— **零命中**。
- `backend-roadmap.md` §1.1 把 privacy-filter 列为**集成"选项"(方案 A/B/C)**,措辞是"建议",**非"已实现"**。
- 模型调用路径:`submitObservation → analyzeObservationResilient → getActiveModelClient`(DeepSeek / packyapi 中转),**送出的是观察原文**。

---

## 2. 真实状况分环节

| 环节 | 真实情况 | 说明 |
|------|:--:|------|
| **数据存储(at rest)** | ✅ 只在本地 | `localStorage`(`experience-os-state-v1`),不上云、不同步 |
| **模型配置 / API Key** | ✅ 只在本地 | `experience-os:model` + `.env.local`(gitignore) |
| **AI 分析(in transit)** | ❌ **原文出设备** | 提炼时观察原文发往云端模型,**无脱敏** |
| **本机隐私过滤层** | ❌ **不存在** | 仅路线图"建议",代码未实现 |
| **中转方(packyapi)侧过滤** | ❓ 无法确认 | 即便有,也是"信任前移给中转方",不在本机、不在掌控 |

> 当前唯一真实的隐私保护 = ① localStorage 不外传(存储层)+ ② **用户自我审查**(不写敏感内容)。**没有第三道"本机过滤"闸。**

---

## 3. 为什么"前端脱敏"不是可选项,而是核心前提

容易把脱敏当"合规附加项"。**错。它和产品核心价值直接挂钩:**

- 最值得做"决策辅助"的数据,恰恰是**敏感的那批**——花钱、健康、人际、职业选择,即"犯错代价高"的核心 niche(见 `painkiller-or-vitamin.md` §6)。
- 若因无脱敏 → 用户不敢写这些 → **不是少了点隐私风险,是丢了核心价值赖以成立的数据**。

> 自我审查是把双刃剑:**降低风险的同时,掏空了高代价决策场景。**
> → **前端脱敏(把 privacy-filter 的正则/熵子集移植到 TS,发请求前剥 PII/密钥)是"让用户敢写敏感决策"的前提**,因此是核心路径,不是锦上添花。

🟡 **[合理推断]** 这条把 `backend-roadmap` 里"方案 A 为主"的优先级,从"隐私合规"升级为"核心价值前提"。

## 5. 风险总结

| 级别 | 风险 |
|:--:|------|
| 🔴 严重 | 用户以为"本机有过滤 / 数据不出设备",据此放心写敏感内容 → 实际原文裸发云端。**认知与现实的落差本身是风险。** |
| 🟡 中等 | 无脱敏 → 用户自我审查 → 高代价决策数据缺失 → 核心价值被掏空。 |
| 🟢 可接受 | 存储层确实本地;BYO-key 直连时内容只到用户自己的模型方(相对可控)。 |

---

## 6. 现状下的诚实表述(对用户该怎么说)

不要说:"数据只在本机 / 已隐私过滤"。
应说:

> - **存储**:你的数据只存在这台浏览器里,不上传、不同步。
> - **AI 提炼**:点击提炼时,这句话会发给云端模型分析;**目前不做脱敏**,请勿写入身份证/银行卡/密钥等敏感信息。
> - **纯离线**:不想任何内容外发,可只用本地兜底引擎(质量较低)。

---

> 收口:**"本机隐私过滤"目前是 0。** 在它落地前,产品的隐私承诺只能停在"存储本地 + 请用户自审";想让用户敢写敏感的高价值决策,**前端脱敏是必须先补的那道闸,且它属于核心价值路径,不是合规附属。**
