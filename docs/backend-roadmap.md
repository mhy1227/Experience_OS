# 后端版本规划

> 日期:2026-06-22 · 状态:规划(当前处于 B0,无后端)
> 关联:产品能力路线图见 `version-roadmap.md`;架构见 `architecture/`;需求全景见 `implementation-status.md`。

## 0. 现状与原则

**当前 = B0:无后端。** 纯前端 + `localStorage` + 浏览器直连模型(Key 在 `.env.local`)。demo / 单人 dogfood 完全够用。

核心原则:
- **按需引入,不按时间**:每个后端阶段都有**明确触发信号**;没出现该需求就不做。
- **保持薄**:后端只做"前端做不了或不该做"的(持 Key、批处理、跨端、多人),不堆框架。
- **本地优先不变**:在必要之前,localStorage 仍是真相源;后端是可选层,不是默认。
- **红线**:① API Key 不进浏览器(这是引入后端的首要驱动);② 后端不擅自扩成团队/知识平台(那是后期、要重定边界)。

## 技术栈(贯穿各阶段)

- **Node + Hono**,部署为 **serverless(Vercel Functions 或 Cloudflare Workers)**:与前端同语言(TS)、免运维、免费档、可长大。
- **复用** 现有 `src/services/analysisContract.ts`(契约层搬到后端共享,Key 也随之上后端)。
- DB 用已有 **SQLPub MySQL**(`db/schema.sql` 的 `exp_` 表,已建)。
- Key/配置走服务端环境变量(不提交)。

## 阶段

### B0 · 无后端(现状)✅
- 前端直连模型 + `.env.local` + localStorage。
- **适用**:答辩 demo、单人使用。
- **够用,不需动。**

### B1 · 模型代理(薄后端)
- **触发**:要开放给别人用 / Key 不能再放在浏览器 / 批量导入想更快更稳。
- **交付**:
  - `POST /api/analyze`(单条)、`POST /api/analyze-batch`(批量,md 导入用);
  - 服务端持 Key、转发模型、**复用契约层**校验降级;
  - 批处理:并行 / 重试 / 限流 / **缓存(同文本不重复花钱)** / 降级。
- **前端改造**:直连改调 `/api`;localStorage 仍为存储。
- **体量**:S–M(一两个端点)。**后端最有价值的第一步**(尤其 `analyze-batch`)。

### B2 · 持久化 / 云同步
- **触发**:要多端 / 数据要云端备份。
- **交付**:observations/rules/insights 读写端点 + **增量同步**(客户端生成 id → 幂等 upsert;靠 `updated_at`)。
- **存储**:SQLPub MySQL(`exp_` 表)。
- **本地仍为主**(离线可用),云为同步层。
- **体量**:M。

### B3 · 向量 / 语义召回
- **触发**:决策辅助要做到语义级(产品 V4)。
- **交付**:服务端生成 embeddings + 存储(MySQL JSON 或向量库)+ 相似检索端点;Key/成本集中在后端。
- **预留**:`exp_rules.embedding` 列已在 schema 里。
- **体量**:M。

### B4 · 账号 / RBAC / 多用户
- **触发**:要给团队/多人用。
- **交付**:账号、鉴权(JWT/session)、**RBAC 权限**、数据隔离(各表加 `user_id`)。
- ⚠️ **产品形态切换**:脱离"纯本地单人",与当前"本地优先 / 不做团队协作"边界冲突——**做之前必须重定产品边界**(是否进入"团队/组织经验"= 产品 V5)。
- **体量**:L。**最后、最重、最需慎重。**

## 触发条件总表(按需,而非按时间)

| 阶段 | 触发信号 | 不做的代价 | 体量 |
|------|----------|------------|:--:|
| B1 模型代理 | 开放他用 / Key 不能在浏览器 / 批量要快 | Key 暴露;批量慢且耗钱 | S–M |
| B2 云同步 | 多端 / 需备份 | 数据仅单机,换设备即丢 | M |
| B3 向量召回 | 决策辅助要语义级 | 召回停留在关键词,精度低 | M |
| B4 账号/RBAC | 多人 / 团队 | 无法多用户、无隔离 | L |

## 非目标(后端不该做)

- 不在必要前用云存储**取代**本地(本地优先;B2 才引入云,且本地仍可用)。
- 不在 B1/B2 就堆账号/团队(那是 B4)。
- 不引入重框架(不为此迁 Nuxt、不上 Spring 等)——**保持 Hono 薄代理**。

## 与产品路线图的对应

| 后端阶段 | 支撑的产品能力 |
|----------|----------------|
| B1 | md 批量导入提速 + Key 安全 |
| B3 | 产品 V4 决策辅助(语义召回) |
| B2 / B4 | 多端 / 团队经验(产品 V5 生态) |

---

> 一句话:**现在不做后端;真要时,Hono+serverless 起一个 `analyze-batch` 薄代理(B1)即可,其余按触发信号逐阶段加,RBAC 最后且需重定边界。**
