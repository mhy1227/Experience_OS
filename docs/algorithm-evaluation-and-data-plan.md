# 算法评估 + 数据 / 冷启动方案

> 日期:2026-06-23 · 状态:规划(支撑性,先于编码)
> 关联:`docs/algorithm-upgrade-plan.md`(总规划)、各 A* 实施方案
> 定位:补总规划里最大的两个洞——**① 怎么证明新算法更好(评估);② 数据少时算法还成不成立(冷启动)。** 不是算法本身,是"让算法可信"的工程。

---

## 0. 为什么这份最该先写

- **没有评估 = 口号**:把"弱关键词"换成"真算法",若不能量化"新的更好",就是自我感觉。
- **没有冷启动方案 = 白做**:TF-IDF/聚类/embedding 都吃数据;真实场景常只有几十条 → 一上线"没效果"。
- 附带红利:**"我建了评估基准"本身是面试强加分项**。

---

## 1. 评估数据集(golden set)

手工标一份小而准的"标准答案",一次投入、长期复用。

| 任务 | 标注内容 | 规模(够用即可) |
|------|----------|------------------|
| 分词 | 句子 + 正确切分 | 50–100 句 |
| 情绪 | 句子 + 正确类别(正/负/中) | 100–150 句 |
| 聚类 | 一组观察 + 每条所属"真规律"编号 | 3–5 簇 × 各 5–10 条 + 噪声若干 |
| 召回 | 查询 + 应召回的规则 id 列表 | 20–30 个查询 |

- 格式:JSON 静态资源,放 `tests/fixtures/`(可 gitignore,避免真实隐私文本上云)。
- 来源:用 `demoWorkData` 风格的合成数据(埋共同根因),**不用真实用户数据**。

---

## 2. 各算法的指标

| 算法 | 指标 | 说明 |
|------|------|------|
| 分词 | **P / R / F1**(按词边界) | 切对的词数 / 标准词数 |
| 情绪 | **准确率 + 各类 F1** | 防"全判中性也高分"的假象 → 看少数类 F1 |
| 聚类 | **纯度 / NMI / 轮廓系数** | 纯度看簇内一致;轮廓看簇是否分得开 |
| 召回 | **recall@k / MRR** | top-k 里有没有命中、命中排多前 |

> 关键:**每个指标都要有"旧关键词基线"并排**(见 §3),只报新分数没意义。

---

## 3. 评估脚本(沿用项目测试纪律)

- 新建 `tests/eval/` 下脚本(node + node:assert 风格,但**输出分数表而非断言**)。
- 每个算法:跑**旧实现**和**新实现**各一遍 golden set,**并排打印** P/R/F1 等。
- 登记一个 `npm run eval` 入口(独立于 `test:evaluation`,因为它是度量不是回归)。
- 产出示例:

```
[分词] 旧(按字): F1=0.41   新(HMM): F1=0.86   ✅+0.45
[召回] 旧(关键词): recall@5=0.52  新(向量): recall@5=0.78  ✅+0.26
```

> **这张并排表 = "上强度"的证据,也是答辩/简历的硬数据。**

---

## 4. 冷启动 / 小数据策略 ⚠️(真风险)

算法吃数据,真实场景常很少。**按数据量分档,定清各档行为:**

| 观察条数 | 本地算法表现 | 策略 |
|----------|--------------|------|
| **N < 20** | TF-IDF/聚类基本无意义 | **以大模型为主**;本地只做分词/脱敏等不吃量的;不强行聚类 |
| **20 ≤ N < 100** | 勉强可用 | 本地聚类作模型的**对照/降级**;阈值放宽 |
| **N ≥ 100** | 本地算法真正发力 | 本地引擎可独立产出,模型作增强 |

- **阈值随数据量自适应**:DBSCAN 的 `minPts`、相似度阈值不写死,按 N 调整(集中到 config)。
- **空结果优雅**:数据不够时明确返回"样本不足,暂不产规律",**不硬凑**(呼应现有 MIN_CLUSTER_SIZE 思路)。
- 诚实结论:**算法不是用来取代模型的,是数据攒够后的离线底座。** 小数据期,模型仍是主力。

---

## 5. 本地 ⊕ 模型 融合编排

定清两套结果怎么共存(否则打架):

- **模型可用**:模型语义聚类为主;本地聚类作**降级备份 + 交叉验证**(两者都圈出的簇,置信更高)。
- **模型不可用**:本地链(分词→TF-IDF→DBSCAN)独立产出,标"本地·低置信"。
- **合并键**:沿用现有 `lawDiscovery` 的"成员重叠优先"逻辑(不受措辞影响)。
- **排序融合**:同一条规律两路都命中 → 提权;只本地命中 → 降权标注。

---

## 6. 参数选择(别写魔数)

| 参数 | 怎么定 |
|------|--------|
| DBSCAN `eps` | **k-distance 图**:排序第 k 近邻距离,找"拐点" |
| DBSCAN `minPts` | 经验 ≈ 维度+1,或按 N 自适应(见 §4) |
| 相似度阈值 | 用 golden set 扫一遍,取 F1 最高点 |

- 全部集中到 `services/algoConfig.ts`(或现有 config),**禁止散落魔数**。

---

## 7. 中文预处理(影响所有下游)

- **停用词表**:去掉"的/了/吗/啊"等无信息词(进 TF-IDF 前)。
- **同义归一**:"对齐会"="对齐会议"="对齐"(小词典映射),提升聚类/召回命中。
- **否定处理**:"没/不/别" + 词 → 影响情绪与语义(分词后按窗口处理)。
- **繁简归一**:统一转简体再处理。

---

## 8. 风险与 YAGNI

- ⚠️ golden set 标注有主观性 → 小规模、关键例为主,断言留弹性。
- ⚠️ 别为评估再造复杂框架(项目有 evaluationEngine 过度工程前车之鉴)——**一个打印分数表的脚本就够**。
- ❌ 不做:自动调参搜索、交叉验证 k-fold、在线 A/B —— 数据量不配,YAGNI。

---

## 9. 简历 / 答辩落点

- "建立离线算法**评估基准**(golden set + P/R/F1/recall@k),量化证明新算法相对关键词基线的提升。"
- "设计**数据量分档**的冷启动策略 + 本地⊕模型融合编排,保证小数据下不强行产出、大数据下离线发力。"

---

> 一句话:**先有这把尺(评估)和这条底线(冷启动分档),再去接算法**——否则要么无法证明有用,要么一上真实少量数据就翻车。
