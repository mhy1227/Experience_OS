# 系统分类 + 用户标签 · 设计分析(面向上班族)

> 状态:分析(待决策 → 实现)
> 日期:2026-06-25
> 关联:`src/types/experience.ts`(`ExperienceCategory`)、`src/services/analysisContract.ts`(`inferCategory` / `categories` 白名单 / 模型 prompt)、`src/services/aiAnalyzer.ts`、`src/pages/index/components/RuleLibrary.vue`(分类筛选)
> 触发:目标用户当前是"上班族",现有 8 类对该人群失衡;同时想引入用户自定义标签。

## 1. 核心结论(先看)

- **两套并存、各管一摊**:**系统分类(category)= AI 自动、单选、固定骨架**;**用户标签(tag)= 用户手动、多选、自由、用于筛选**。这是业界标准的 IA 分工,且**不破"AI 自动结构化,不是带标签的笔记"的卖点**(结构仍由 AI 给)。
- **系统分类按上班族微调**(加 3 类、砍 1 类),但**不为"工作"细分** —— 工作内部颗粒交给用户标签。
- **标签做成可筛选(非纯装饰)+ 自动补全防滥用**。

## 2. 业界怎么分(依据)

通用信息架构:
- **分类/文件夹**:单个、结构性、回答"放在哪",撑骨架,相对固定。
- **标签**:多个、属性性、回答"关于什么",横向关联,灵活。
- 主流建议**混用**:分类管骨架 + 标签管灵活维度;但**标签滥用会乱(tag sprawl)**,需自动补全/建议标签克制。

来源:Evernote《Folders vs Tags》、Adnovum《Categories or Tagging(taxonomy/IA)》、Medium《How to Structure Notes — Categories, Tags, Folders》。

## 3. 系统分类:面向上班族的新增角度(本文重点)

### 3.1 现状与问题
现有 8 类:`饮食 / 购物 / 出行 / 运动 / 工作 / 生活 / 偏好 / 其他`。对上班族:
1. **「工作」是一个大桶**:该人群 ~80% 经验在工作(对齐/评审/排期/沟通/汇报…),全挤一类,颗粒太粗。
2. **缺高频生活域**:`学习成长`、`健康`、`理财` 没有对应类 → 这类经验只能落「其他」(演示数据实测如此)。
3. **「偏好」含糊**:很难界定,易空置,价值低。

### 3.2 关键判断:别为"工作"细分
分类粒度对**核心价值影响很小** —— 规律发现是**跨 category 的语义聚类**(A2/A3)、找经验/决策建议靠**语义召回**,都不依赖分类;category 主要用于**规则库筛选 + 冷启动分桶兜底**。
> 所以把「工作」拆成会议/沟通/项目…只会让枚举臃肿、收益低。**工作内部颗粒应交给用户标签**(`#会议 #沟通 #向上管理 #排期`)。

### 3.3 两个方案

**方案 A(纯增量,最稳):** 保留 8 类,**新增** `学习成长 / 健康 / 理财` → 11 类。
- 优点:零数据迁移(不动旧类),最安全。
- 缺点:11 类偏多;`偏好` 含糊仍在。

**方案 B(再平衡,推荐):** → `工作 / 学习成长 / 健康 / 理财 / 出行 / 饮食 / 购物 / 生活 / 其他`(9 类)
- **加**:`学习成长`、`理财`;
- **并**:`运动` → 归入 `健康`;`偏好` → 归入 `生活`;
- 优点:均衡覆盖上班族高频域,枚举不臃肿,语义更清。
- 缺点:`运动`/`偏好` 是**删类**,需迁移历史数据(把旧 category 重映射到新类),且是不可逆动作。

> 建议 **方案 B**(更干净、更贴人群),迁移用一次性映射 `运动→健康`、`偏好→生活` 即可。若怕迁移风险,退方案 A。

## 4. 用户标签设计

| | 谁定 | 性质 | 用途 |
|---|---|---|---|
| category | **AI 自动** | 单选、结构骨架 | 聚类 + 筛选,**保持固定枚举** |
| **tags(新增)** | **用户手动** | 多选、自由文本 | **筛选/组织**(横切维度),不影响 AI 分类/聚类口径 |

- **数据**:`ExperienceRule` 新增 `tags?: string[]`(用户标签,与 AI 自动产的 observation.tags 区分:这是规则级、用户级)。
- **UI**:RuleCard 上增删标签 chip;RuleLibrary 加「按标签筛选」。
- **防滥用**:标签输入**自动补全**(从全库已有标签去重建议),避免同义标签泛滥。
- **定位红线**:标签是**纠错/补充/筛选层**,不是录入主流程 —— 录入仍是"一句话 → AI 自动结构化"。

## 5. 改动面(精确触点)

## 6. 风险 / 不可逆 / 取舍

- **删类不可逆**:`运动/偏好` 删除需数据迁移;迁移脚本要幂等、跑前可回退(导出备份)。→ 故保守可选方案 A(纯增)。
- **模型 prompt 不同步 = 新类形同虚设**:模型仍按旧类归,新类只能靠本地 `inferCategory` 兜底。→ 改类必同步 prompt。
- **标签滥用**:不配自动补全/建议,会退化成一堆同义碎标签。→ 自动补全是必配项。
- **产品哲学**:标签只能是"补充/筛选",一旦变成"录入要自己标",就回退成标签笔记 → 稀释卖点。

## 7. 推荐与分期

1. **先做系统分类调整(方案 B)**:加 `学习成长/健康/理财`、并 `运动→健康`/`偏好→生活`;同步 prompt + inferCategory + 一次性迁移 + 演示数据。**这是对上班族最直接的优化。**
2. **再做用户标签**:`rule.tags` + 编辑 + **可筛选** + 自动补全,定位"纠错/补充"。
3. **不做**:为"工作"细分系统分类(用标签替代);放开 category 为完全自由文本(失去聚类骨架)。
