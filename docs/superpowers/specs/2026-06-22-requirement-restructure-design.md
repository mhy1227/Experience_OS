## 1. 背景与方向迁移

当前产品由「轻量录入闭环 + 极重评估体系」组成,重心严重偏向"单条规则的可验证性证明"(evaluationEngine 1714 行 + 46 条评估用户故事),但**缺失新方向的灵魂能力:跨多条记录发现规律、决策时主动召回提醒**。

```
现在:  一句话录入 → 单条规则 → 【庞大复测/采用/评估体系】(≈70% 复杂度,过度工程化)
目标:  一句话录入 → 经验提炼(真AI) → 跨记录【规律发现】→【决策辅助提醒】
```

核心差异化:日记沉淀事件,Experience OS 沉淀规律;日记帮回忆,我们帮决策。

## 3. 增 / 删 / 改清单

### 新增(进 5 天 MVP)
- **[增] M3 规律发现**:跨记录聚合面板,显示高频问题 Top3 / 成功模式。
- **[增] M4 决策辅助**:新观察命中历史高频模式时弹"你过去 N 次类似记录"提醒。
- **[增] M5 API Key 配置入口**:设置页填 Provider + Key,本地存储,演示模型/自配模型切换。
- **[增] 提炼三段式 Prompt 第一段(经验提取)**;归类/规律发现 Prompt 后置。

### 砍 / 降级(关键:砍过度工程化,代码保留不删,仅从主路径摘除)
- **[降] 复测矩阵(确认/边界/对照/扩展四槽位)** → 高级面板或隐藏。
- **[砍] 采用门槛 / 可重复性画像 / 样本独立性 / 协议执行检查单 / 协议合规画像 / 维护回归 / 恢复观察期** → MVP 不展示。
- **[降] 复测三态** → 保留为规则卡上轻量"这条还准吗"反馈,不再驱动整套评估推导。
- **[降] CSV 导出** → 降级;JSON 导出保留(作为"导出我的经验资产",契合本地优先叙事)。

### 改写
- **[改] 定位文案**:"个人经验自动结构化系统" → "让经历变成决策资产 / 让每次踩坑产生复利"。
- **[改] 输入接纳范围**:扩到工作/沟通/投资/情绪等高共鸣事件。
- **[改] 演示数据**:重做成"能体现系统在学习"的一组(见 §6)。
- **[改] 默认 AI 入口**:内置演示模型开箱即用(本地引擎兜底),高级用户再配 Key。

## 4. 数据层调整

现有 7 张 `exp_` 表能覆盖:经验闭环、成功率、时间趋势、按类目频次。缺口与调整:

- **[改] `exp_observations` 加结构化情绪列**:`sentiment VARCHAR(16)`(positive/neutral/negative,可聚合、建索引)+ 可选 `sentiment_score`、`emotion`。**不要塞进 tags(JSON)**,否则"过去 90 天负面占比"无法高效聚合。
- **[增] `exp_insights` 表(规律/洞察,核心新增)**:语义独立于 rule(rule 是用户经验单元,insight 是系统跨记录归纳的统计派生单元,与 rules/observations 多对多)。关键列:`type`(pattern/trend/correlation/anomaly)、`title`、`summary`、`confidence`、`time_window`、`metrics JSON`、`status`、`generated_by`、`evidence_rule_ids JSON`、`evidence_observation_ids JSON`。
- **[增] `exp_model_configs` 表**:`provider`、`model`、`base_url`、`api_key_ref`(**密文/本地引用,绝不存明文**)、`is_active`、`is_default`。
- **[增,P1 可选] `exp_recall_events`**:记录决策辅助召回事件,用于回溯"决策辅助是否有用"。
- **[原则] 本地优先 vs 云 MySQL**:云 MySQL 从"唯一真相源"降为"可选同步/备份",本地(IndexedDB / SQLite)为主存储,`exp_` schema 双端同构;**API Key 永不上云**;演示模型产物纯本地。

## 5. AI 流水线

**可直接复用的地基(无需重写)**:
- `ObservationModelClient` 接口(`modelAnalysisAdapter.ts`)——唯一模型抽象点,任何厂商实现 `completeJson` 即可接入。
- `analyzeObservationWithModel()`——已含"调用 + 解析 + 失败降级"骨架。
- `OBSERVATION_ANALYSIS_PROMPT`——第一段 Prompt,已含防注入 + 方向约束。
- `normalizeModelAnalysis` / `enforceAnalysisContract` / `assertValidAnalysis`——方向一致性校验 + 降级链,模型无关,接真模型后照样防"幻觉成规律"。

**5 天最小实现(三件)**:
1. 写一个统一 client 工厂 `createModelClient(config)`(OpenAI/DeepSeek 同格式一份实现;Claude 后补)。
2. store 接入点(`experience.ts` 现走 `analyzeObservation` 处):模型优先,失败回退本地 `analyzeObservation`。
3. 双模式(演示 Key / 用户 Key)+ Key 存 localStorage(导入导出排除 Key)+ 错误可见反馈。

三段式 Prompt:**第一段(经验提取)必做**;第二段(经验归类)、第三段(规律发现)后置(依赖数据量,且校验最复杂)。

## 6. 演示(Hero Demo)

**一句话设计:不演"记录",演"被看穿"。**

| 时间 | 屏幕 | 话术(精简) |
|------|------|------|
| 0:00–0:30 | PPT:"为什么你总在同一个坑栽?" | "我们以为在进步,其实在重复犯错,只是看不见模式。" |
| 0:30–1:10 | 经验列表,预置 30–40 条横跨 90 天工作观察,快速滚动 | "这是我 90 天随手写的工作片段,从没整理。" |
| 1:10–1:50 | 现场输入 1 条:"这周又因需求中途改方向,加班两天白干" → 生成策略卡 | "今天又记一条,看起来跟上次没关系吧?" |
| 1:50–2:40 ★ | 点「扫描我的 90 天」→ 归因卡片:**"38 条工作记录中,80% 负面结果指向同一根因:目标不一致/中途变更。"** + 3 条证据时间线 | (停顿)"我以为是时间管理、是同事配合——AI 说 80% 是同一个原因。" |
| 2:40–3:10 | 卡片给出可复用规则 + 决策建议:"下次接需求前先 5 分钟确认目标与验收,可规避历史 80% 返工。" | "它不只说哪错了,还给了下次的动作。这就是经验变资产。" |
| 3:10–3:30 | 三大原则页(Experience/Local/Model Agnostic) | "数据全在你电脑上,AI 是分析者不是拥有者。" |

**顿悟瞬间定义**:把用户自己没串起来的 N 条分散记录,归因到一个出乎意料的共同根因,并量化成百分比。震撼来自:跨时间 + 出乎意料 + 量化。

**演示数据**:预置 ~35 条工作观察(2026-03~06),其中 12–15 条埋同一根因(不同表述/时间)、~15 条干扰项、~5 条正向规律。每条带日期 + 结果方向 + 根因标签(预置写死,不必真做 NLP)。

**演示中坚决不碰**:复测矩阵/采用门槛/维护回归/协议合规画像、地图页、CSV 导出、(按选定方案乙)90 天归因走规则统计而非真模型。
