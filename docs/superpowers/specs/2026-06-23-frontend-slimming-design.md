## 3. 目标 / 非目标

**目标**
- 首屏显著变短:打开即见"录入 + 关键产出",评估细节默认收起。
- 落实 CLAUDE.md"评估体系降级隐藏":评估矩阵默认不展开,可在「高级」/展开区按需查看。
- 不删任何评估功能、不改业务逻辑、不动数据模型。

**非目标**
- 不重写 evaluationEngine 逻辑(只改"展示与否")。
- 不引入路由框架(保持单页;用折叠/「高级」开关即可)。
- 不在本次强制拆完所有组件(组件化见 §6,可分期)。

## 4. 方案 A(推荐):折叠 / 下沉评估矩阵

**A1. `RuleCard` 默认精简**
- 默认只显示:标题、kind 徽标(策略/避坑/待观察)、可复用度、结论、推荐行动、适用条件、证据记录、反馈按钮(有用/待观察/不准确)。
- 评估详情(重复评估 / 复测协议 / 采用门槛 / 槽位 / 画像 / 复测输入)收进卡内 `<details>`「展开评估详情」,默认折叠。
- 影响:`最新策略卡` 和 `规则库` 里每张卡同时变短。

**A2. 规则库的"评估工作台"下沉到「高级」**
- 220–882 的评估工作台整体包进 `v-if="showAdvancedPanel"`(已有该开关),规则库 tab 默认只列精简规则卡。
- 「高级」开关已存在,只需把工作台真正收到它后面。

**收益**:页面短 60%+,首屏清爽;只折叠不删,完全对齐红线;风险低(纯展示层)。

## 5. 方案 B(顺手):录入区收敛

- `快速记录` + `批量导入` + `分析模式条` 合并为一个"录入"卡:首屏只留**一个输入框 + 主按钮**;「批量导入 / md 导入 / 载入演示 / 导出 / 清空 / 扫描」收进"更多"折叠或一行图标。
- 让 tab 内容(经验/规律库)更早出现在首屏。

## 6. 模块边界(按能力层切)+ 组件化(方案 D,单独立项)

> 原则:**模块边界对齐产品能力层**(蓝图:采集 → 提炼 → 关联 → 验证 → 决策 → 组织),而非"现在屏幕长什么样"。这样未来新能力是"加并列模块",不是"重切现有"。否则 V4 决策一来就得重排。

### 6.1 顶层模块(稳定,对齐能力层)

| 模块 | 能力层 | 含 | 未来演化 | 边界是否变 |
|---|---|---|---|---|
| **录入 InputModule** | 采集 | 快速记录 + 批量 + md;次要操作(演示/导出/清空)收"更多" | V5 加飞书/微信/插件渠道适配器,汇入同一管线 | 内部长子组件,**边界不变** |
| **状态条 StatusBar** | — | 引擎 v1 / 评估数 / 闭环(一行,非一屏) | 基本不变 | 不分 |
| **规则产出 RuleLibrary** | 提炼+验证 | RuleCard(精简)+ 评估详情折叠 | V3 验证深化 → `RuleEvaluationPanel` 内部长大 | 内部长,边界不变 |
| **规律产出 LawLibrary** | 关联(V2) | 规律库 + 洞察 + 复盘 | 拆 `LawLibrary/InsightList/PeriodicReview` | 内部长,边界不变 |
| **浏览 ExperienceList/Map/Timeline** | — | 经验/地图/时间轴 | 稳定 | 不变 |
| **高级 EvaluationWorkbench** | 验证 | 评估工作台(下沉折叠) | V3 在此深化 | 内部长 |
| **决策 DecisionSurface** | 决策(V4) | **暂无** | V4 落地 → **新增并列模块/tab**(录入时风险提醒/场景召回) | **新增 sibling** |
| **组织** | 组织(V5) | **暂无** | V5 → 新增组织经验面 | **新增 sibling** |

**结论:这次按上表切,以后 V4/V5 = 加模块,不动老模块;老模块只向内裂子组件,对外接口/位置不变。**(YAGNI:V4/V5 没来,不先建空壳。)

### 6.2 组件化拆分(方案 D,单独 PR 逐个提)

- `components/InputModule.vue`(录入)/ `StatusBar.vue`(状态条)
- `components/RuleCard.vue`(**从 142 处 `h()` 命令式渲染提成 SFC 模板**)+ 子组件 `RuleEvaluationPanel.vue`(评估折叠)
- `components/EvaluationWorkbench.vue`(高级)
- `components/LawLibrary.vue` + `InsightCard.vue`(已有)+ `PeriodicReview.vue`
- `components/ExperienceList.vue` / `ExperienceMap.vue` / `Timeline.vue`
- `index.vue` 收敛为"壳":接 store + tab 切换 + 组合上述模块。

目标:`index.vue` 从 4068 行降到数百行;每个组件 <300 行、props/events 接口清晰。详尽的前端优化盘点见 [前端优化分析](../../architecture/frontend-optimization-analysis.md)。

## 7. 分期与验证

- **阶段 1(A)**:RuleCard 折叠 + 评估工作台下沉「高级」。验证:typecheck + `test:evaluation` 全绿;Playwright 真机量首屏高度明显下降、评估详情可展开、「高级」可见完整工作台、规律库/反馈/复测功能不回归。
- **阶段 2(B)**:录入区收敛。验证同上 + 录入/导入/扫描链路真机回归。
- **阶段 3(D)**:组件化拆分(单独 PR/分支),逐个组件迁移 + 每步 typecheck/测试,保证零行为变更。

## 8. 风险

- `RuleCard` 是 `<script>` 内渲染函数(非 SFC),折叠改动需小心 DOM 结构;先加折叠、阶段 3 再提 SFC。
- 评估工作台事件(复测/采用/修订)下沉后仍需可达——「高级」开关必须显眼,且默认折叠不影响数据写入。
- 纯展示层改动,不碰 store/evaluationEngine 逻辑,回归面小。

## 9. 建议结论

先做 **A**(性价比最高、对齐红线、低风险),**B** 顺手;**D** 作为后续重构单独立项。本期先出此文档,实施与否由产品方决定。
