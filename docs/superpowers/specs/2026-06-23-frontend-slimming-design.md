## 1. 问题

H5 单页(`index.vue`)整体纵向过长:一屏一屏往下滚很久才到主要内容。单文件 **4068 行**(2026-06-23),且所有区块纵向堆叠、无折叠/路由。

## 2. 诊断(实测行号)

页面从上到下,**始终可见区**(在 tab 之外,任何 tab 都压着):

| 区块 | 行号 | 始终可见 |
|---|---|---|
| 头部 stats + 快速记录(输入框 + 样例按钮 + 一排操作) | 1–110 | ✅ |
| 批量导入(textarea + 按钮 + md 导入) | 111–153 | ✅ |
| 分析模式条 | 154–165 | ✅ |
| **最新策略卡 = 完整 `RuleCard`** | 167–179 | ✅ |
| tab 栏 + 「高级」开关 | 181–198 | ✅ |

tab 内面板:经验(200)、规则库(219–882)、地图(883)、时间轴(901)、规律发现(920–998)。

**🔴 头号元凶 —— 评估体系(evaluationEngine)UI 全量铺开:**

1. **`最新策略卡` 是一张完整 `RuleCard`,永远显示在 tab 之上**(167 行),而 `RuleCard`(渲染函数在 `<script>` 内)内嵌了**整个评估矩阵**:重复评估、复测协议、采用门槛、槽位 0/4、可重复性/一致性画像……(渲染函数标记见 1731 行「重复评估」、1852 行「复测协议」)。→ **每次打开页面,首屏就压着一大坨评估细节。**
2. **规则库 tab 是约 660 行的"评估工作台"**(219–882),含 ~25 个子区:采用决策 / 采用门槛 / 可重复性画像 / 一致性画像 / 维护槽位待办 / 未解决维护回归 / 回归恢复复核 / 复测矩阵 / 复测协议 / 协议执行 / 反例边界 / 修订草案 / 决策变化 / 复测覆盖度 / 样本质量 / 复查提醒 / 复测计划 / 召回候选……全量展开。

**这正是 CLAUDE.md 反复要求"降级隐藏而非删除"的过度工程部分**,但当前是**全程展开**,贡献了页面 60%+ 的长度。已有 `showAdvancedPanel`(「高级」开关,193 行)但评估细节并未真正收到它后面。

**🟡 次要:** 录入区(快速记录 + 批量导入 + 分析模式)占满首屏,要滚很久才到 tab 内容。

## 3. 目标 / 非目标

**目标**
- 首屏显著变短:打开即见"录入 + 关键产出",评估细节默认收起。
- 落实 CLAUDE.md"评估体系降级隐藏":评估矩阵默认不展开,可在「高级」/展开区按需查看。
- 不删任何评估功能、不改业务逻辑、不动数据模型。

**非目标**
- 不重写 evaluationEngine 逻辑(只改"展示与否")。
- 不引入路由框架(保持单页;用折叠/「高级」开关即可)。
- 不在本次强制拆完所有组件(组件化见 §6,可分期)。

## 4. 方案 A(推荐):折叠 / 下沉评估矩阵

**A1. `RuleCard` 默认精简**
- 默认只显示:标题、kind 徽标(策略/避坑/待观察)、可复用度、结论、推荐行动、适用条件、证据记录、反馈按钮(有用/待观察/不准确)。
- 评估详情(重复评估 / 复测协议 / 采用门槛 / 槽位 / 画像 / 复测输入)收进卡内 `<details>`「展开评估详情」,默认折叠。
- 影响:`最新策略卡` 和 `规则库` 里每张卡同时变短。

**A2. 规则库的"评估工作台"下沉到「高级」**
- 220–882 的评估工作台整体包进 `v-if="showAdvancedPanel"`(已有该开关),规则库 tab 默认只列精简规则卡。
- 「高级」开关已存在,只需把工作台真正收到它后面。

**收益**:页面短 60%+,首屏清爽;只折叠不删,完全对齐红线;风险低(纯展示层)。

## 5. 方案 B(顺手):录入区收敛

- `快速记录` + `批量导入` + `分析模式条` 合并为一个"录入"卡:首屏只留**一个输入框 + 主按钮**;「批量导入 / md 导入 / 载入演示 / 导出 / 清空 / 扫描」收进"更多"折叠或一行图标。
- 让 tab 内容(经验/规律库)更早出现在首屏。

## 6. 方案 D(单独立项):组件化拆分 index.vue(4068 行)

> 治"难维护"(与"视觉太长"是两件事),工程量较大,建议**作为后续重构单独推进**,不与 A/B 混做。

建议拆分(每个职责单一、接口清晰、可独立测试):
- `components/QuickRecord.vue`(快速记录 + 样例 + 生成规则)
- `components/BatchImport.vue`(多行 + md 文件导入)
- `components/RuleCard.vue`(从 `<script>` 渲染函数提成 SFC;含"评估详情"折叠子组件 `RuleEvaluationPanel.vue`)
- `components/EvaluationWorkbench.vue`(规则库的评估工作台整块)
- `components/LawLibrary.vue`(V2 规律库)+ `InsightCard.vue`(已是组件)+ `PeriodicReview.vue`
- `components/ExperienceList.vue` / `ExperienceMap.vue` / `Timeline.vue`
- `index.vue` 收敛为"壳":状态接入 + tab 切换 + 组合上述组件。

目标:`index.vue` 从 4068 行降到数百行;每个组件 <300 行。

## 7. 分期与验证

- **阶段 1(A)**:RuleCard 折叠 + 评估工作台下沉「高级」。验证:typecheck + `test:evaluation` 全绿;Playwright 真机量首屏高度明显下降、评估详情可展开、「高级」可见完整工作台、规律库/反馈/复测功能不回归。
- **阶段 2(B)**:录入区收敛。验证同上 + 录入/导入/扫描链路真机回归。
- **阶段 3(D)**:组件化拆分(单独 PR/分支),逐个组件迁移 + 每步 typecheck/测试,保证零行为变更。

## 8. 风险

- `RuleCard` 是 `<script>` 内渲染函数(非 SFC),折叠改动需小心 DOM 结构;先加折叠、阶段 3 再提 SFC。
- 评估工作台事件(复测/采用/修订)下沉后仍需可达——「高级」开关必须显眼,且默认折叠不影响数据写入。
- 纯展示层改动,不碰 store/evaluationEngine 逻辑,回归面小。

## 9. 建议结论

先做 **A**(性价比最高、对齐红线、低风险),**B** 顺手;**D** 作为后续重构单独立项。本期先出此文档,实施与否由产品方决定。
