# Experience OS 定位分析与边界界定

> 日期:2026-06-23 · 状态:分析(供决策)
> 起因:考虑「飞书多维表格接入 + 每天定时扫描任务」,需先确认产品定位是否清晰。
> 关联:`docs/version-roadmap.md`(产品路线图)、`docs/backend-roadmap.md`(后端 B0–B4)、`CLAUDE.md`(产品方向/红线)

---

## 0. 一句话结论

定位**概念**没错:「经验 → 决策资产」是一个真实存在、基本空白的品类位置。
定位**风险**真实存在,但不在"知识管理 vs 单纯记录"这条线上,而是:

> **产品事实上仍站在「记录 / 知识管理」区,而把脱身的全部筹码押在尚未建好的 V3(验证)/ V4(决策)上。** 此时若提前接飞书 + 扫任务,会在差异化未被证明前,把产品同时往「采集层」和「任务管理」两个相邻品类拉,反而让"到底是什么"更模糊。

---

## 1. 坐标校准:四个相邻品类

把容易混淆的相邻品类摊开,定位问题就清楚了:

| 品类 | 代表产品 | 核心价值 | 时间指向 |
|------|----------|----------|----------|
| 单纯记录 | 日记 / Flomo / 备忘录 | 捕获 + 回看 | 过去(事件) |
| 知识管理(PKM) | Obsidian / Notion / Logseq | 组织 + 关联 + 检索 | 沉淀(知识) |
| **经验决策(本产品声称占位)** | **(市场基本空白)** | **提炼教训 → 辅助决策** | **过去 → 未来** |
| 任务管理 | Todoist / 飞书多维表格 | 安排 + 追踪 | 未来(待办) |

**关键判断:** 用户担心的"介于知识管理和记录之间",描述的是*现状*;产品*声称*的格子(经验决策)其实是第三格,与前两者不同。所以问题不是"站错格子",而是"还没真正站进去声称的格子"。

---

## 2. 核心风险:差异化兑现得太晚

| 能力 | 当前状态 | 落在哪个品类 |
|------|----------|--------------|
| V1 提炼(AI 打标签 / 结构化) | ✅ 已建 | **记录 / 知识管理**(文档自承"接近二次打标签") |
| V2 基础(category/tag 聚类) | ✅ 已建 | **知识管理**(自动聚类) |
| V2 升级(Law 语义聚类 + 生命周期) | 🟡 分支未合并 | 开始向"经验决策"靠拢 |
| V3 验证(可信度 / 成功率) | 🟡 降级隐藏 | **经验决策**(差异化所在) |
| V4 决策召回("这门课你买过6次,完成率25%") | 🟡 基础版 | **经验决策**(杀手时刻) |

**确定性标注:**
- 🟢 已建成的 V1 + V2 基础,在*用户行为感知*上与"AI 笔记 + 自动聚类"难以区分。
- 🟡 让产品脱离记录 / 知识管理区的筹码,几乎全押在 V3 + V4——而这两块最不完整。
- 🔴 **整个定位是否成立的命门:用户会不会"回来做决策"?** 若用户只来"记"、从不回来"决策",产品就坍缩成又一个 AI 笔记。目前这是假设,无数据。

> 结论:**不是定位骑墙,是差异化交付太晚。** 现阶段事实上在"记录/知识管理"区,声称的"决策资产"还停在路线图上。

---

## 3. 对「飞书多维表格 + 每天定时扫任务」的专项审查

### 3.1 飞书接入 = 加码"采集层"(顺序倒置)
- 🟢 产品自定义:"采集是入口,不是产品;智能层才是护城河"。
- 🟢 `backend-roadmap` 把飞书明确划在 **V5(最后、最重、最慎,做前必须重定边界)**。
- ⚠️ 现在提前 = **在智能层价值被证明之前,先建采集 / 生态层**。护城河没挖,先修引水渠。

### 3.2 "扫任务" = 拽入第四个品类(任务管理)—— 最危险
- 🔴 Experience OS 的内核是**经验(过去的教训)**,不是**任务(未来的待办)**。两者是不同的域。
- 🔴 一旦开始处理 todo,产品同时横跨 记录 + 知识管理 + 任务管理 **三个格子**,"经验决策"这个唯一差异点被稀释到不可见。
- ⚠️ 口语里的"扫任务**啥的**"——"啥的"二字就是 scope creep 的信号灯。

### 3.3 "每天定时自动扫描" 与产品哲学的张力
- 🟢 红线:"AI 辅助提炼,**不替代思考**"。
- ⚠️ 全自动定时扫描,若产出"替你想好的结论" → **违背哲学**;若产出"提示你回来做一次决策复盘" → **强化哲学**。同一机制,两种命运,必须先定清楚它输出什么。

### 3.4 不可逆性
- 🟡 飞书接入一旦上线、用户依赖多维表格做数据源,**回退成本很高**(数据归属、同步语义、用户预期全绑死)。
- ⚠️ 而它带来的仍是"更多记录入口",不是"更强决策"——**高成本、低差异化、难回退**的组合,典型该缓。

---

## 4. 边界界定:把「经验决策」这个格子写死

为防止定位被稀释,明确**什么进、什么不进**:

### ✅ 属于 Experience OS(进)
- **观察 / 经验 / 教训**:已发生的、含因果或可复用判断的口语片段。
- **跨记录的规律**:同根因不同表述的归并(Law)。
- **基于历史的决策辅助**:在新决策点召回相关经验。
- **经验的验证与演化**:复发、是否已改进、可信度。

### ❌ 不属于 Experience OS(不进 / 推迟)
- **未来待办 / 任务安排 / 进度追踪** —— 这是任务管理域,**坚决不做**(即使飞书表里有任务字段,也只抽取其中的"经验/复盘"语义,不接管任务本身)。
- **纯事件流水 / 日记** —— 那是记录域;本产品只收"含判断的观察",不收"今天吃了什么"。
- **通用知识库 / 第二大脑 / RAG 平台** —— 知识管理域,V1 非目标已明令推迟。
- **团队 / 组织协作** —— V5,做前必须重定边界。

### ⚠️ 灰色地带的处理原则
- 飞书多维表格**可以作为采集入口**,但只抽取其中**经验性内容**(复盘、踩坑、规律),**不导入任务条目本身**。
- 定时扫描**只用于发现经验规律 / 提示决策复盘**,**不用于任务提醒 / 进度催办**。

---

## 5. 风险总结

| 级别 | 风险 |
|:--:|------|
| 🔴 严重 | 差异化价值(决策资产)押在最不完整的 V3/V4;"扫任务"可能引入任务管理域,稀释唯一差异点。 |
| 🟡 中等 | 飞书提前到 V5 之前,顺序倒置——先建采集层再证明智能层。 |
| 🟢 可接受 | V1 暂时"像笔记"本身不致命,**前提是 V3/V4 能及时兑现**。 |

---

## 6. 建议的优先级(供决策,非承诺)

> 本节为决策建议,非实现计划。

1. **先证明差异化,再扩采集**:把筹码先押在 V3/V4 的最小可感版本("回来做一次决策"的杀手时刻),让产品真正站进"经验决策"格子。
2. **飞书按原规划留在 V5**:确需提前,也只作**只读采集入口**,且只抽经验性内容,不接管任务。
3. **定时扫描定性为"决策复盘提醒",不是"任务催办"**,守住"不替代思考"红线。
4. **"任务"明确列入不做清单**:写进 `version-roadmap.md` 的 V1/V2 非目标,防止 scope creep。

---

> 决策口径:**接飞书 / 扫任务之前,先回答一个问题——用户会不会回来做决策?** 这个问题没有正向信号之前,任何采集层扩张都是在为一个未被验证的内核修管道。
